#### 什么是RPC
1. Remote Procedure Call 远程过程调用
   该调用过程涉及通信协议，远程调用的核心设计思想是：调远程方法像调本地方法一样简单
   优点：数据传输体积小、低延迟、上手便捷
   rpc通信协议可以是http
2. `RPC`面向应用程序的通信 应用之间通信 性能更好
   rpc只是定义了不同服务之间数据通信的一个规范 本身不是具体实现
   如何理解定义规范：比如日志定义规范，springcloud定义规范。rpc是定义规范 其具体实现可能是http
3. `rpc`与`进程间通信`：rpc是进程间通信的一种方式 能够直接调用另一个jvm进程的Java方法
   http是访问服务器资源，而rpc Java方法也是资源


#### Dubbo概述
1. `分布式系统形态`：
   多个OS进程组成分布式系统，进程之间内存隔离，不能直接调用方法，必须走网络通信（RPC/HTTP）
   分布式靠多个隔离的 JVM 进程跨网络协作
2. `Dubbo定位`：
   Dubbo是一个`分布式系统组件`，`Dubbo`是基于rpc的分布式框架，dubbo框架能够实现`RPC`
3. `Dubbo形态`：
   1. 无独立进程形态，Dubbo 是内嵌在每个业务 JVM 里的 RPC 通信组件，自身不能单独作为进程运行
      不管是 Provider 还是 Consumer：
      Dubbo 只是项目引入的依赖框架，和 Controller、Service 代码跑在同一个 JVM、同一个进程中
      (像`Nacos`、`SkyWalking`都是分布式系统中的独立进程，而Dubbo不是独立进程)
   2. Provider：JVM 内启动 Netty Server（监听 20880），对外提供 RPC；
      Dubbo 内置 Netty 服务端（Provider）：监听独立端口（默认 20880），接收外部 RPC 请求
      Consumer：JVM 内启动 Netty Client，发起远程调用；
      Dubbo Netty 客户端（Consumer）：维持和远端 Provider 的 TCP 长连接
      同一 JVM 进程可同时兼具 Provider + Consumer 双重角色


#### Dubbo作用
1. `实现RPC`(最核心作用)
2. `服务治理`
   1. 服务注册与发现
   2. 负载均衡：随机 / 轮询 / 最小活跃数 / 一致性哈希
   3. 集群容错策略：失败重试、快速失败、失败广播、降级
3. `序列化`
   网络只能传输二进制，Java 对象必须序列化后才能通过 TCP 发送
   网络影响：序列化字节越少，单包网络负载越低，网卡、交换机转发更快，减少拥塞概率
   Dubbo 支持多种序列化，都是把对象转为字节数组写入 TCP 缓冲区
   1. Hessian2：默认，跨语言兼容性好
   2. Fastjson2/Kryo：二进制紧凑，网络传输字节更小，减少带宽占用
   3. Protobuf：Triple 协议专用

#### Dubbo有哪些通信协议
1. `http协议`：
   1. 各层协议：
      应用层：HTTP/1.1、HTTP2，基于 Servlet / Jetty / Netty
      传输层：TCP
   2. 约定数据格式：json
   3. 特点：跨语言友好，浏览器、前端、第三方系统可直接调用
   4. 缺点：报文头大，性能低于 dubbo 协议
   5. 适用：对外开放接口、异构系统对接
2. `Dubbo协议`(默认协议)：
   1. 各层协议：
      应用层：自定义的二进制 RPC 协议(性能远高于 HTTP/1.1)
      传输层：TCP
   2. 约定数据格式：二进制
   3. 特点：轻量、高性能、长连接、NIO 异步，适合服务间内网高频调用
      Dubbo 协议基于 TCP 长连接，对比 HTTP 短连接优势
      HTTP 每次交互都要三次握手 + 四次挥手，大量 RPC 调用网络开销巨大
      Dubbo 消费者启动时和 Provider 建立少量持久 TCP 长连接，反复复用，省去频繁握手，减少网络 RTT 耗时
      TCP 天然问题：字节流无边界 -> Dubbo 自定义二进制报文结构解决粘包拆包
      Dubbo 支持多种序列化
   4. 适用：Java 微服务内部互通（Dubbo 生态首选）
   5. Dubbo 协议报文结构（应用层自定义帧，解决 TCP 半包）
   6. IO 模型：Netty NIO（传输层 Socket 调度）
3. `grpc`
   1. 各层协议：
      应用层：HTTP2
   2. 序列化：Protobuf
   3. 特点：谷歌标准、跨语言、流式通信、序列化紧凑
   4. 适用：跨语言微服务、大文件流式传输、云原生场景
4. `Triple 协议`(Dubbo3 新增)(Dubbo 自定义 RPC 协议)：
   1. 各层协议：
      应用层：HTTP2
   2. 序列化：Protobuf
   3. 特点：同时支持：流式调用、普通 RPC、跨语言、网关友好
      兼容 Dubbo2 服务，云原生标准协议，Dubbo3 推荐默认协议
      基于 HTTP2，兼容浏览器、Go/Python 跨语言
      通信模型：长连接，单连接多路复用，类似 HTTP2 优势
5. `Hessian`(Dubbo 自定义 RPC 协议)


#### rpc调用流程
一次rpc 涉及TCP/IP 四层
1. `发起`：从上到下发包（不断加包头）
   1. 应用层：Dubbo 协议封装接口参数 -> 原始业务报文
      产生要传输的数据 比如请求参数 发消息给谁
   2. 传输层：TCP 协议加 TCP 头（确认源端口、目标端口）
      可靠传输（TCP）/ 快速发送（UDP）
   3. 网络层：IP 协议加 IP 头（确认本机 IP、服务端 IP）
   4. 网络接口层：以太网头（本机 MAC、交换机 MAC），转为电信号发出
      把二进制数据通过网线 / 光纤 / 网卡，在同一个局域网内传输
2. `接收`：从下往上拆包
   1. 剥掉以太网帧头 -> 交给 IP 层
   2. 剥掉 IP 头 -> 交给 TCP 层
   3. 剥掉 TCP 头 -> 交给 Dubbo 应用层
   4. 最终执行远程方法


#### rpc基于dubbo协议通信流程完整网络链路
1. Provider 启动：创建 ServerSocket 绑定 IP: 端口，内核监听 TCP 端口（传输层）
2. 消费者启动：通过 Nacos 获取 Provider IP，发起 TCP 三次握手建立长连接
3. 调用接口：消费者封装 Dubbo 报文，序列化对象写入 TCP 发送缓冲区
4. 操作系统内核：TCP 滑动窗口分段，IP 协议路由，经交换机 / 网关传到服务端机器
5. Provider 内核：网卡接收二进制字节流，Netty NIO 线程读取缓冲区
6. 解码器先读 16 字节包头，通过长度字段截取完整报文，反序列化为 Java 对象执行业务
7. 返回结果：同样封装 Dubbo 响应报文，沿原有 TCP 连接回传
8. 服务下线：TCP 四次挥手断开长连接


#### Dubbo 协议核心网络设计取舍
1. `为什么选 TCP 而非 UDP`
   UDP 无连接、不可靠、不保证有序，RPC 场景要求请求必须送达、响应不乱序；
   Dubbo 业务需要可靠交付，因此全部协议底层都基于 TCP（牺牲少量性能换取可靠性）。
2. `长连接设计的网络收益`
   规避频繁三次握手（每次握手 3 个 RTT 往返延迟）；
   减少操作系统 TCP 连接表、文件句柄占用；
   TCP 连接会持续维护滑动窗口、拥塞状态，长连接下传输效率更稳定。
3. `消息 ID 多路复用解决的网络痛点`
   若一个请求一条 TCP 连接，高并发会创建数万 Socket，操作系统内核维护大量 TCP 控制块，造成 CPU、内存、端口资源耗尽；
   单连接多请求复用 TCP 通道，大幅降低网络资源消耗。
4. `固定长度包头解决 TCP 底层固有缺陷`
   TCP 是流式协议，操作系统内核只负责搬运字节，不区分业务报文边界；
   若无长度字段，Netty 读取缓冲区时会出现：多条请求粘在一起、一条请求被拆成多次接收，业务直接报错，这是所有 TCP 私有协议必须处理的网络底层问题。
5. `和 Nacos 注册中心的网络配合`（完整微服务网络链路）
   注册：Provider 发起 TCP 连接 Nacos，上报自身 IP + 端口（应用层注册数据，走 TCP）；
   发现：消费者 TCP 请求 Nacos，拉取服务节点列表；
   真正 RPC：消费者直连 Provider TCP 长连接，不经过注册中心，避免注册中心成为网络瓶颈；
   分层网络隔离：服务发现流量、业务 RPC 流量分离，各司其职
