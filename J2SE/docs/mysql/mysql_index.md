#### 索引概述
1. `索引`是在`存储引擎`中实现的，索引就是为了提高查询性能，相当于书的目录，但也有区别  
2. 表数据在文件系统中存储的基本单位是数据页 16KB
   数据页中存储表的几条数据 行格式 行格式记录表的下一条数据的信息  
3. 文件系统中 页之间 表数据之间 都不一定是物理连续的 是逻辑连续
4. `索引是存储引擎用于快速找到数据的一种数据结构`，索引底层数据结构主要是：b+树、哈希 相当于书的目录  
   `b+树`极其简化后就是`二叉搜索树`  
   `索引是一种排好序的数据结构`
5. 有了索引mysql存储引擎可以更快速查找数据，这里也涉及数据结构的知识点
   有了索引就减少了查找数据时磁盘的io，时间复杂度低了
   mysql开发工程师设计索引就是为了加快mysql的速度 让mysql更好用


#### 创建索引
1. `方式`：
   1. 建表时
      1. `普通索引`：INDEX idx_name (name),
      2. `唯一索引`：UNIQUE INDEX uk_phone (phone),
      3. `联合索引`：INDEX idx_name_age (name, age)
   2. 表已存在
      1. `普通索引`：CREATE INDEX idx_user_name ON t_user(name);
      2. `唯一索引`：CREATE UNIQUE INDEX uk_user_phone ON t_user(phone);
      3. `联合索引`：CREATE INDEX idx_user_name_age ON t_user(name, age);
      4. `全文索引`：CREATE FULLTEXT INDEX ft_article_content ON t_article(content);
2. `说明`：
   给一个字段创建索引：为这个字段排好序 构建索引b+树 回车耗费时间和空间  
   插入/删除数据时 索引也会跟着调整 为了快可以事先删除索引 之后再加上索引  
   索引在添加有些约束时会隐式的创建 比如主键约束 唯一约束 外键约束等
3. `不同索引含义`：
   1. `普通索引` 是最基本的索引，没有任何限制 添加了索引的字段的值可以为空
   2. `唯一索引` 字段的值可以=null，唯一索引列的值必须唯一
   3. `主键索引` 是特殊的唯一索引，一张表只能有一个主键，不能为空
   4. `联合索引` 注意与 联合主键区分开


#### 删除/查看索引
1. 删除索引：DROP INDEX idx_user_name ON t_user;
2. 查看索引：SHOW INDEX FROM t_user;


#### 隐藏索引
不使用索引了 但是此时 索引的结构还会随增删改查变化


#### 索引设计原则
1. `哪些字段适合/不适合加索引`
   1. `去重` 得排序 所以加索引好(表数据量小 且对某字段去重查询操作频繁 给该字段加索引合适)
      去重字段如果不加索引，会把要查询的数据全查到内存进行一一比对去重
      如果加索引 遍历索引即可挑选出 去重数据
2. `尽量选择过滤性更好的字段当索引`
3. `(重点)索引失效场景`
    1. `全值匹配`：where条件 等值 创建全的联合索引
    2. `最左前缀原则` 但如果查询的字段有联合索引字段 则会使用索引 结论：MySQL可以为多个字段创建索引，
       一个索引可以包括16个字段。对于多列索引，过滤条件要使用索引必须按照索引建立时的顺序，依次满足，
       一旦跳过某个字段，索引后面的字段都无法被使用。如果查询条件中没有使用这些字段中第1个字段时，多列(或联合)索引不会被使用。
       关注`多列联合索引`的b+树是怎么排序的 create index(a,b,c) -> 排序规则：先排a，a相同的这些数据排b，b相同的再排c 如果a没有相同的就不用排b，c了
       因此查询时如果没有使用a字段，则索引失效，总之了解了多列索引的排序规则 就可以理解索引失效的场景了。(重点)
    3. `计算` `函数` `类型转换`(类型不匹配 还是使用了函数)会导致索引失效 函数为什么会导致索引失效：索引列本身有序 加了函数之后不一定有序了 所以索引失效
    4. 范围条件右边的列索引失效 大于小于 where a=1 and b>10 and c=99 c索引失效
       在 a=1,b>10 筛选出来的一堆数据里，索引上的 c 是无序的 -> 没法用索引快速找 c=99，只能内存遍历过滤，所以 c 索引失效
    5. !=对于普通索引失效 根据数据分布决定索引是否失效 数据 90% 都!=18，优化器认为走索引还要回表、随机 IO，成本高于全表，索引失效
    6. is not null索引可能失效 数据分布决定
       表大部分是 NULL，只有少量非空 -> 走索引（筛选少量数据，索引划算）
       表绝大部分数据非空、只有一点点 null -> 优化器放弃索引、全表扫描
       原因：is not null = 取绝大部分数据，走索引要大量回表随机 IO，全表顺序读更快。(就是因为主键索引id是有序的 而回表的id是无序的 所以回表会大量随机io)
    7. like 以%开头索引失效 b+树不知道怎么查了
       但是like 以%结尾索引不失效 根据前缀会划分范围 -> 索引有效
    8. OR 前后存在非索引的列，索引失效 根据这些去体会b+树的查询方式
    9. 数据库和表的字符集统一使用utf8mb4 否则会索引失效


#### 索引类型
1. 从数据库物理存储角度：`聚簇索引(主键索引)`、`非聚簇索引(非主键索引)` 主键索引存完整数据
2. 从功能角度：`普通索引` `唯一索引` `主键索引` `全文索引(全文检索)`
3. 从作用字段个数角度：`单列索引` `联合索引(组合索引)`


#### 索引b+树的整体结构
`哪个是b树 b+树 要会区分：`b+树只有叶子节点存储表真实数据 非叶子结点只存储目录页数据。所以比b树更矮胖 io次数更少
1. `索引(b+树)`
2. `索引节点(索引节点是数据页) 关注页结构`
    1. 叶子节点：所有的叶子节点都包含了实际的数据记录或者指向实际数据记录的指针，并且叶子节点之间通过双向指针或者链表的形式连接起来，形成一个有序的序列，方便进行范围查询。
    2. 非叶子节点：非叶子节点主要用于索引和导航，它们存储了`键值`以及`指向子节点的指针`，用于引导查找操作向下进行。
        1. 键值：非叶子节点存储了一些键值，这些键值用于确定数据的范围和查找路径。键值的选择通常是根据数据的某个特定属性或者索引列来确定的，它们在节点中按照从小到大的顺序排列
3. `叶子节点数据页里面是真实数据 关注行格式`
4. B + 树是平衡树，这意味着树中任意一个叶子节点到根节点的路径长度都是相等的


#### hash索引和b+树索引区别
(底层数据结构、增上改查性能、磁盘利用率、磁盘随机/顺序访问等方面)
1. hash索引是哈希表结构 hashmap底层也是这个结构 b+树是多路平衡树的结构
2. hash索引这种数据结构决定了hash索引在等值查询有很好的性能 对于范围查询和排序hash索引性能比较差  
   b+树索引在范围查询和排序上更加高效 因为b+树是有序的
3. hash索引插入和删除比较简单 通过hash函数确定索引位置插入即可 删除也是链表所方便的  
   b+树插入删除需要维护树的平衡性 需要节点的拆分和合并 更复杂
4. hash索引对于磁盘利用效率不高 哈希索引数据分布是随机的 会导致磁盘的随机访问  
   b+树节点是有序存储的 有利于磁盘的顺序访问 减少io次数
